How it works in practice.

Four agents. Each is a specialist. They don't know each other. They don't trust each other. They transact anyway, using receipts as proof.

Scene 1: Sentinel computes
Sentinel has proprietary satellite data. No one else has it. It runs inference on a trusted hardware enclave (AWS Nitro). The enclave signs the output with its private key. That signature is unforgeable proof that Nitro ran the code. Sentinel wraps that into a receipt: 
Receipt ID:      0x5b4c3cab...
Model:           sentinel-v2
Proving backend: tee-nitro-v1
Provider:        Sentinel [d7311d09...]
Output hash:     0xd47bddbe...
Signature:       [real Ed25519, verifiable by anyone]
The receipt is permanent. Cryptographically. Anyone can verify it.
Scene 2: MedGraph builds on it
MedGraph needs Sentinel's output. It doesn't trust Sentinel. So it verifies the receipt first: checks the signature, verifies the proof, checks the computation happened. Then MedGraph uses Sentinel's output as input to its own model. Creates a new receipt: 
Receipt ID:      0xdb0381ce...
Model:           medgraph-v3
Provider:        MedGraph [5194dfea...]
Parent receipts: [Sentinel's receipt ID]
Output hash:     0x8f2c4d19...
Signature:       [MedGraph's Ed25519]
Now this receipt includes a cryptographic link to Sentinel's receipt. The chain is tamper-evident.
Scene 3: Nexus orchestrates all three
Nexus can't do any of these tasks itself. It's the orchestrator. It calls Sentinel, MedGraph, and Regulatory. Uses all three outputs to compute investment strategy. Creates a receipt with three parents: 
Receipt ID:      0x9d2e3feb...
Model:           nexus-strategy
Provider:        Nexus [9d2e3feb...]
Parent receipts: [Sentinel, MedGraph, Regulatory]
Output hash:     0x12ab45cd...
Signature:       [Nexus's Ed25519]
The provenance DAG is now: Sentinel MedGraph Regulatory \ | / \ | / \ | / Nexus (depth 1) Every edge is a hash. Modify Sentinel's receipt and Nexus's receipt breaks. The chain is the audit trail.
Scene 4: A fund buys Nexus's output
Fund wants Nexus's strategy. Doesn't know Nexus. Doesn't need to. Fund verifies: — Nexus's signature (Ed25519) ✓ — Nexus's parents (Sentinel, MedGraph, Regulatory) ✓ — Each parent's signature and proof ✓ — The entire chain is intact ✓ Fund buys for $150. Creates a transfer record: 
Receipt ID:      0x9d2e3feb...  (same)
From key:        Nexus [9d2e3feb...]
To key:          Fund [48b79151...]
Price:           $150
Seller signature: [Nexus's Ed25519 on transfer]
The transfer is appended to the log. Permanently. Witnessed by 3 independent servers.
Scene 5: Royalties flow backward
When Fund buys Nexus's receipt, royalties cascade back through the DAG automatically. Nexus gets: $147.75 (provider royalty 5% + seller remainder) Sentinel gets: $0.75 (parent royalty 3%, split among parents) MedGraph gets: $0.75 Regulatory gets: $0.75 No platform. No accounting. Protocol math. Every time Fund resells to someone else, the cascade happens again. Sentinel keeps earning forever. Not because anyone manages it. Because the receipt carries the royalty terms.

Why this is God

You have proof. You have provenance. You have payment. All three together, in one data structure.

No platform took a cut. No escrow held the money. No middleman verified anything. Just cryptographic math and an append-only log.

Run it yourself:

git clone https://github.com/tesseravcr/tesseravcr.git
cd tesseravcr
python3 tessera-py/demo.py

Real Ed25519 signatures. Real SHA-256 hashes. Real Nitro enclave attestation. Real ZK proofs. You'll see every step.
